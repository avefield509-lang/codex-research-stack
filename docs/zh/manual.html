这份说明书主要管什么

这份说明书只解释当前这个公开仓库：它是什么、它适合谁、第一次怎么开始、项目跑起来以后看什么，以及它和未来 app 应该怎么分工。

它不负责解释你的私人本地环境细节，也不负责解释未来 app 的内部实现。所以你在这里看到的，是一份公开可复用版的用户手册。

先用三句话说清楚

第一，这个仓库不是聊天机器人本身，而是一套“研究工作怎么组织”的规则和模板。

第二，它的重点不是多装工具，而是让研究项目从一开始就更有条理。

第三，它想解决的是研究里最容易失控的部分：任务怎么开头、项目怎么推进、材料怎么留痕、写作前哪些东西必须先检查。

你可以把它理解成一个研究工作台：你先说你要做什么，系统先告诉你该怎么开始，项目做大之后，它会把过程整理成可以回看的结构。

你不用先懂这些词

  • research-autopilot：接待台。你先说需求，它先判断这是什么任务、应该怎么开始。
  • research-team-orchestrator：项目经理。只有项目变复杂时，它才会出来安排角色、步骤和检查点。
  • gate：检查口。某一步没过关，项目就不应该继续往前推。
  • Zotero：正式文献库。
  • Obsidian：长期研究笔记和知识沉淀的地方。

你只需要记住一句话：先让系统告诉你该怎么开始，再决定项目要不要变复杂。

它适合谁

  • 做文献综述的人
  • 做平台案例、网页材料或社媒案例研究的人
  • 做计算社会科学项目的人
  • 正在写论文、返修或整理投稿材料的人

如果你只是偶尔问一个很小的问题，它也能用。但它最有价值的时候，是你已经意识到：这个任务不像一条聊天能说完，它更像一个项目。

这套公开版真正由什么组成

  • Codex：前台对话和执行入口
  • research-autopilot：先判断任务、解释路线
  • research-team-orchestrator：把复杂任务整理成项目协作
  • 项目文件：留下规则、状态、发现和检查记录
  • 外部工具：Zotero、Obsidian、浏览器证据链

换句话说：Codex 负责“你怎么进来”，autopilot 负责“先走哪条路”，orchestrator 负责“项目怎么排”，项目文件负责“以后怎么回看”，外部工具负责“文献、笔记和证据各放在哪里”。

第一次使用，只做这三件事

  1. 把仓库拉下来
  2. 创建一个示例项目
  3. 先让系统解释路线，不要急着执行
git clone https://github.com/avefield509-lang/codex-research-stack.git
cd codex-research-stack
python .\scripts\init_research_project.py --path ".\examples\demo-project" --route-hint "general-research"
pwsh -ExecutionPolicy Bypass -File ".\scripts\init-research-project.ps1" -Path ".\examples\demo-project"

第一句最推荐这样说：

@research-autopilot 这是一个新的研究项目。
主题是：...
问题是：...
材料或数据是：...
预期产出是：...
先解释应该怎么开始，再决定要不要升级成项目型协作。

如果你完全不想折腾命令

那你现在只需要记住一句话：先说清楚你的主题、问题、材料和想要的产出，让系统先给出路线，再开始做。

这就是最核心的用法。也就是说，哪怕你还不会跑命令、看脚本、管目录，也可以先用这套思路开始。

新项目怎么开始

  1. 选一个项目目录
  2. 初始化项目骨架
  3. 看几个最重要的文件
  4. 让系统先解释路线

先看下面几个文件就够了：

  • AGENTS.md：项目规则
  • research-map.md：项目地图
  • findings-memory.md：已确认的发现
  • logs/project-state/current.json：当前项目状态

很多人第一次看到这些文件会紧张。其实可以把它们理解成：项目规则、项目地图、已确认的发现和当前状态。

对大多数普通研究者来说，平时最常用的其实只有两件事：开项目和查状态。

项目为什么会进入多智能体协作

小任务先走单一流程。只有项目变大、步骤变多、需要分工和复核时，才会进入多智能体协作。

真正的多智能体协作不是“一个人分饰几角”，而是把项目分工、复核和交接真正写进文件。

  • 不同角色有不同身份
  • 不同角色有各自的输入
  • 不同角色有各自的输出位置
  • 至少有明确的复核关系
  • 项目里会留下可回看的痕迹

项目跑起来后，你最该看什么

项目一旦跑起来，最重要的不是“它调用了多少工具”，而是三件事是否清楚：

  1. 它现在在做哪一步
  2. 下一步该谁来做
  3. 有没有哪一步其实还不能继续

所以最重要的一句话是：不要先盯着工具，先看任务判断和项目状态。

最值得先看的文件

  1. AGENTS.md
  2. research-map.md
  3. findings-memory.md
  4. material-passport.yaml
  5. evidence-ledger.yaml
  6. logs/project-state/current.json
  7. logs/quality-gates/pipeline-status.json

如果你只想知道项目现在怎么样，这几份文件基本够了。

这套系统适合哪些场景

  • 文献综述：提出问题、收集文献、核验引用、形成综合稿
  • 社媒或平台案例：采集浏览器可见内容、保留来源、准备后续编码材料
  • 计算社会科学项目：把文献、材料、分析、写作和复现串起来
  • 写作与投稿：草稿、引文检查、写作质量和投稿材料整理

它和 Zotero、Obsidian 这些工具是什么关系

一句话说：Zotero 管正式文献，Obsidian 管长期笔记，项目目录管运行中的状态和痕迹。

对于平台与网页案例，这套仓库默认更保守：优先看浏览器可见内容，明确保留来源，把“我看到了什么”和“我怎么解释它”分开。

当前仓库和未来 App 是什么关系

最容易理解的说法是：

  • 当前仓库是“规则和说明书”
  • 未来 app 是“更顺手的界面”

仓库是源头，app 是界面。

这个公开仓库不包含什么

它不是私人工作站镜像。它不会公开私人文件、本地 trust 状态、私钥、云端运维材料和本地运行输出。

如果出问题，先看哪里

  1. 先看项目卡在哪：logs/project-state/current.json
  2. 再看项目在做什么：research-map.md 和 findings-memory.md
  3. 再看是不是有检查没过：logs/quality-gates/pipeline-status.json
  4. 最后才怀疑仓库本身：运行 python .\scripts\validate_research_stack.py
