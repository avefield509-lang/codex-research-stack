Codex 研究工作台

让 Codex 里的研究项目保持清楚。

Codex Research Stack 更像一张研究工作台。它先帮你判断任务该怎么开头， 再帮你把项目状态、关键检查和材料沉淀放到能回看的位置上， 而不是让所有东西都挤进一条很长的对话里。

这个仓库主要想解决什么

很多研究自动化工具在“已经知道该怎么做”之后才有用。真正容易出错的是更前面：任务判断、项目推进、检查点和材料交接。

先把任务说清楚

系统会先告诉你这是什么类型的工作、建议怎么开始，而不是直接把工具全调起来。

项目不再像长对话

项目型任务会变成一个能回看的工作区，谁在做、交给谁审、卡在哪一步都能看见。

检查点摆在台面上

引文、写作质量和复现检查会显式出现，不再靠“默认应该没问题”往前推进。

先用三分钟看懂它

快速 Demo 展示一个项目型任务如何从用户提示，进入路线解释、dispatch，再被 reviewer gate 检查。

Route

Dispatch

把工作拆给角色

项目扩展之前，producer、reviewer、产出和状态先变成可见结构。

Handoff

材料不要混成一团

文献、项目文件、笔记、证据和最终产出分别沉淀，不塞进同一条聊天。

如果你暂时不想跑命令，先看 3 分钟快速 Demo。

现在只保留一份说明书

你不需要先把整个系统学会。公开说明现在都收在一份总说明书里，你只要按自己的目的跳到对应章节就可以。

我想先试一下直接跳到最短、安全的上手路径。 我想知道项目跑起来后发生什么看项目从 route 到 dispatch 会留下哪些可读痕迹。 我想直接开一个项目看一个项目怎么从零开始，以及最该先看哪些文件。
我想看适合什么场景看文献综述、平台案例、计算社会科学和写作场景。 我想看工具怎么接理解 Zotero、Obsidian、脚本和证据采集各自处在什么位置。 我想先理顺未来 app 的位置看当前仓库负责什么，未来 app 又该建立在哪一层。 我想一次看全打开总说明书，从目录直接跳到需要的章节。
