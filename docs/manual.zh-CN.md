## 1. 这份说明书帮你做什么

这份说明书的作用，是让你最快看懂并试用 Codex Research Stack：

- 它是什么

<a id="what"></a>
## 2. 先用三句话说清楚

第一，这个仓库不是聊天机器人本身，而是一套“研究工作怎么组织”的规则和模板。  
第二，它的重点不是多装工具，而是让研究项目从一开始就更有条理。  
第三，它想解决的是研究里最容易失控的部分：任务怎么开头、项目怎么推进、材料怎么留痕、写作前哪些东西必须先检查。

你可以把它理解成一个研究工作台：

- 你先说你要做什么
- 系统先告诉你该怎么开始
- 项目做大之后，它会把过程整理成可以回看的结构

<a id="glossary"></a>
## 3. 你不用先懂这些词

第一次看这个仓库时，很多人会被术语劝退。其实可以先用人话理解：

- `research-autopilot`
  - 你可以把它当成“接待台”
  - 你先说需求，它先判断这是什么任务、应该怎么开始

- `research-team-orchestrator`
  - 你可以把它当成“项目经理”
  - 只有项目变复杂时，它才会出来安排角色、步骤和检查点

- `gate`
  - 你可以把它当成“检查口”
  - 某一步没过关，项目就不应该继续往前推

- `Zotero`
  - 正式文献库
  - 真正要写进论文的正式引用，最后应该回到这里

- `Obsidian`
  - 长期笔记和知识沉淀的地方
  - 它更像你的研究笔记系统，不是正式文献库

所以你一开始根本不需要理解所有技术名词。  
你只需要记住一句话：

**先让系统告诉你该怎么开始，再决定项目要不要变复杂。**

<a id="who"></a>
## 4. 它适合谁

这套仓库最适合下面几类人：

- 做文献综述的人
- 做平台案例、网页材料或社媒案例研究的人
- 做计算社会科学项目的人
- 正在写论文、返修或整理投稿材料的人

如果你只是偶尔问一个很小的问题，它也能用。  
但它最有价值的时候，是你已经意识到：**这个任务不像一条聊天能说完，它更像一个项目。**

<a id="components"></a>
## 5. 这套系统真正由什么组成

如果不用技术词，你可以把这套系统拆成五层：

- `Codex`
  - 负责前台对话和执行入口
- `research-autopilot`
  - 负责先判断任务、解释路线
- `research-team-orchestrator`
  - 负责把复杂任务整理成项目协作
- `项目文件`
  - 负责留下规则、状态、发现和检查记录
- `外部工具`
  - 例如 Zotero、Obsidian、浏览器证据链

换句话说：

- `Codex` 负责“你怎么进来”
- `autopilot` 负责“先走哪条路”
- `orchestrator` 负责“项目怎么排”
- `项目文件` 负责“以后怎么回看”
- `外部工具` 负责“文献、笔记和证据各放在哪里”

<a id="start"></a>
## 6. 第一次使用，只做这三件事

第一次不要想着把整个系统学会。只做这三件事就够了：

### 1. 把仓库拉下来

```powershell
git clone https://github.com/avefield509-lang/codex-research-stack.git
cd codex-research-stack
```

### 2. 先创建一个示例项目

```powershell
python .\scripts\init_research_project.py --path ".\examples\demo-project" --route-hint "general-research"
```

如果你在 Windows 上，也可以直接用：

```powershell
pwsh -ExecutionPolicy Bypass -File ".\scripts\init-research-project.ps1" -Path ".\examples\demo-project"
```

### 3. 不要急着执行，先让它解释

第一句最推荐这样说：

```text
@research-autopilot 这是一个新的研究项目。
主题是：...
问题是：...
材料或数据是：...
预期产出是：...
先解释应该怎么开始，再决定要不要升级成项目型协作。
```

这一步的作用，不是“显得专业”，而是避免一开始就走错方向。

<a id="new-project"></a>
## 9. 新项目怎么开始

开一个新项目时，不需要一口气理解所有文件。  
最稳的顺序只有四步：

1. 选一个项目目录
2. 初始化项目骨架
3. 看几个最重要的文件
4. 让系统先解释路线

初始化之后，你不用到处翻。先看下面几个文件就够了：

- `AGENTS.md`
  - 这个项目的规则写在这里

- `research-map.md`
  - 这个项目到底要解决什么、目前怎么拆

- `findings-memory.md`
  - 已经确认了什么、哪些发现值得保留

- `logs/project-state/current.json`
  - 现在项目进行到哪一步、卡在哪一步

如果这四个文件都在，项目骨架基本就搭起来了。

### 你可以怎么理解“项目骨架”

很多人第一次看到这些文件会紧张，以为自己要管理一堆技术文件。

其实可以把它理解得非常朴素：

- `AGENTS.md` = 项目规则
- `research-map.md` = 项目地图
- `findings-memory.md` = 已确认的发现
- `project-state` = 项目当前状态

也就是说，它只是把原本散落在聊天、脑子和临时笔记里的东西，强制整理到了固定位置。

### 平时最常用的其实只有两件事

对于大多数普通研究者，日常最常用的不是一堆脚本，而是：

1. 开项目
2. 查状态

也就是：

- 初始化项目
- 说清楚任务
- 看项目地图
- 看发现记录
- 看当前状态

<a id="real-multi-agent"></a>
## 10. 项目为什么会进入多智能体协作

很多人第一次听到“多智能体”，会以为系统一开始就会自动分出很多角色。  
其实不是。

更准确的说法是：

- 小任务先走单一流程
- 只有项目变大、步骤变多、需要分工和复核时，才会进入多智能体协作

所以多智能体不是为了“看起来高级”，而是因为项目已经复杂到需要：

- 分工
- 交接
- 复核
- 留下痕迹

### 什么才算真正的多智能体协作

这个问题很重要，因为很多系统会把“像多个角色说话”包装成多智能体。

在这个仓库里，真正的多智能体协作必须满足：

- 不同角色有不同身份
- 不同角色有各自的输入
- 不同角色有各自的输出位置
- 至少有明确的复核关系
- 项目里会留下可回看的痕迹

如果只是同一段回复里换几个口吻，那不算。

你可以把真正的多智能体协作理解成：

**不是“一个人分饰几角”，而是真的把项目分工、复核和交接写进文件。**

<a id="live-project"></a>
## 11. 项目跑起来后，你最该看什么

项目一旦跑起来，最重要的不是“它调用了多少工具”，而是下面这三件事是否清楚：

1. 它现在在做哪一步
2. 下一步该谁来做
3. 有没有哪一步其实还不能继续

这个仓库把项目过程分成三层来看：

### 第一层：任务判断

由 `research-autopilot` 负责。

它先回答：

- 这到底是什么任务
- 应该先走哪条路线
- 有没有必要把它升级成一个项目

### 第二层：项目协作

如果任务已经不再是一个小问题，而变成了一个项目，`research-team-orchestrator` 才会出来。

你可以把它理解成“把项目整理成一个可回看的工作区”：

- 谁负责哪一段
- 哪一步需要复核
- 现在卡在哪

### 第三层：执行

到了这一步，才是文献核验、证据采集、分析、写作和导出这些实际工作。

所以最重要的一句话是：

**不要先盯着工具，先看任务判断和项目状态。**

<a id="use-cases"></a>
## 13. 这套系统适合哪些场景

### 文献综述

它适合把下面几步连起来：

- 提出综述问题
- 收集候选文献
- 核验正式引用
- 整理成综合稿

### 社媒或平台案例研究

它适合把下面几步连起来：

- 读取浏览器可见内容
- 保留证据来源
- 把采集和解释分开
- 为后续编码和分析准备材料

### 计算社会科学项目

它适合把下面几步连起来：

- 文献
- 材料
- 分析
- 写作
- 复现

重点不是“什么都能做”，而是整个项目可以在同一套结构里推进。

### 写作、返修与投稿

它适合把下面几步连起来：

- 草稿写作
- 引文核验
- 写作质量检查
- 回复材料和投稿材料整理

<a id="integrations"></a>
## 14. 它和 Zotero、Obsidian 这些工具是什么关系

一句话说：

- Zotero 管正式文献
- Obsidian 管长期笔记
- 项目目录管运行中的状态和痕迹

### Zotero

Zotero 是正式文献边界。

也就是说，真正要写进论文参考文献里的东西，最后应该回到 Zotero。

### Obsidian

Obsidian 是长期知识层。

也就是说，项目里沉淀出来的可复用笔记、想法和综合内容，可以放到 Obsidian。

### 浏览器可见证据

对于平台与网页案例，这套仓库默认更保守：

- 优先看浏览器可见内容
- 明确保留来源
- 把“我看到了什么”和“我怎么解释它”分开
- 把采集和解释分成两步
- 留下足够的上下文，让别人能回看证据从哪里来

### 运行环境工具

未来 app 可以检查当前电脑是否已经具备：

- Python
- Git
- Codex Home
- Zotero
- Obsidian
- 浏览器工具链

缺一个工具，不应该把整个系统搞崩；最多只是某些功能先不用。

<a id="repo-and-app"></a>
## 15. 当前仓库和未来 App 是什么关系

最容易理解的说法是：

- 当前仓库是“规则和说明书”
- 未来 app 是“更顺手的界面”

仓库负责：

- workflow rules
- schemas
- validators
- templates
- examples
- 公开说明文档

未来 app 负责：

- onboarding
- dashboard
- 任务入口
- 项目可视化界面

所以未来 app 不是另起一套规则，而是建立在当前仓库上面。

产品层面的关系很简单：仓库定义工作流包，app 让这个工作流更容易启动、查看和继续。

如果用一句最简单的话概括：

**仓库是源头，app 是界面。**

<a id="troubleshooting"></a>
## 16. 如果出问题，先看哪里

很多问题其实不用一上来就排脚本。  
先按这个顺序看：

### 1. 项目到底卡在哪

先看：

- `logs/project-state/current.json`

### 2. 项目现在处在哪一阶段

再看：

- `research-map.md`
- `findings-memory.md`

### 3. 是不是有检查没过

再看：

- `logs/quality-gates/pipeline-status.json`

### 4. 如果你怀疑工作流本身坏了

最后才跑：

```powershell
python .\scripts\validate_research_stack.py
```

你可以把这理解成一条很简单的排查顺序：

- 先看项目状态
- 再看项目地图
- 再看检查口
- 最后才怀疑仓库本身
