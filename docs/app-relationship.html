Repository and future app

One source of truth, one interface layer.

This repository should remain the public source of truth for the research workflow. The future app should sit on top of it as a guided interface, not as a competing system with its own private rules.

Back to home Read architecture

This repository owns

  • workflow definitions
  • catalogs, schemas, and templates
  • validators and public scripts
  • docs, examples, and public assets

The future app should own

  • onboarding and project dashboards
  • task-entry forms and status views
  • visual route explanations
  • guided project creation and environment views

What the app should read instead of redefining

The app should derive its UI model from the repository's public contracts, especially the catalog, schemas, templates, examples, and validators. It should not silently fork the workflow logic.

What the app should never become

  • a second source of truth for route logic
  • a hidden fork of the check and gate rules
  • the only way to understand how the stack works
