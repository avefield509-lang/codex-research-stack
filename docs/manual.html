What this manual helps you do

This manual is the fastest way to understand and try Codex Research Stack: what it is, who it is for, how to begin, and what to inspect during a live project.

It also explains how Zotero, Obsidian, evidence capture, project files, and the future app fit into one research workflow.

In three sentences

First, this repository is not the chatbot itself. It is the workflow layer around research work in Codex.

Second, its goal is not to add more tools. Its goal is to make research projects easier to start, inspect, and continue.

Third, it focuses on the part where research work usually becomes messy: how a task begins, how a project stays readable, when weak work should be blocked, and how references, notes, traces, and outputs stay separated.

Terms you do not need to worry about at first

  • research-autopilot: the front desk. You describe the task, and it explains how work should begin.
  • research-team-orchestrator: the project manager. It only appears when work grows into a real project.
  • gate: a checkpoint. If something is not good enough yet, the project should not move forward.
  • Zotero: the formal reference library.
  • Obsidian: the longer-lived note and knowledge layer.

If you remember only one thing: ask the system to explain the route before it starts doing work.

Who this is for

  • researchers doing literature reviews and evidence synthesis
  • computational social scientists working across text, platform data, networks, and reproducibility
  • researchers collecting browser-visible platform or web evidence
  • writers preparing drafts, revision packs, and submission-ready materials

It can still help with smaller tasks. But it becomes most useful when you already realize that the task is turning into a project.

What the stack is actually made of

  • Codex: the front door for conversation and execution
  • research-autopilot: decides what kind of task this is and how work should begin
  • research-team-orchestrator: turns larger work into structured project coordination
  • project files: write down rules, status, findings, and checks
  • external tools: things like Zotero, Obsidian, and browser-visible evidence capture

First use: only do these three things

  1. clone the repository
  2. create a demo project
  3. ask for route explanation before execution
git clone https://github.com/avefield509-lang/codex-research-stack.git
cd codex-research-stack
python .\scripts\init_research_project.py --path ".\examples\demo-project" --route-hint "general-research"
pwsh -ExecutionPolicy Bypass -File ".\scripts\init-research-project.ps1" -Path ".\examples\demo-project"

A good first prompt is:

@research-autopilot This is a new research project.
The topic is: ...
The question is: ...
The materials or data are: ...
The expected outputs are: ...
Explain the route first, then decide whether this should become project-type multi-agent work.

If you only have three minutes

Open the quick demo if you want to understand the project before running anything.

  • 3-minute quick demo
  • demo-prompt.md: the user request
  • route-explanation-card.md: the route explanation
  • .codex/dispatch/demo-run.yaml: the agent dispatch
  • logs/project-state/current.json: the project state
  • gate.literature-producer.json: the reviewer gate

The demo shows a task being routed, turned into explicit agent roles, and stopped by a reviewer gate when formal citation verification is still missing.

If you do not want to deal with commands yet

Then remember this sentence: state your topic, question, materials, and expected outputs clearly, and ask the system to explain the route before it starts doing work.

That is the core usage model, even before you learn scripts or directories.

How a new project begins

  1. create a project folder
  2. initialize the scaffold
  3. inspect the core files
  4. let the system explain the route before execution

Start with:

  • AGENTS.md = project rules
  • research-map.md = project map
  • findings-memory.md = what is already known
  • logs/project-state/current.json = current status

For most researchers, daily use mostly comes down to two things: starting a project and checking the current state.

Why a project turns into multi-agent work

Small tasks stay simpler. Only larger project work turns into multi-agent coordination.

That happens when the project really needs division of work, handoff, review, and written traces.

It only counts as real multi-agent work when the project leaves behind separate identities, separate inputs, separate outputs, reviewer links, and inspectable traces.

What to look at during a live project

  1. what stage the project is in
  2. who owns the next step
  3. what still blocks progress

So the practical rule is: do not start by staring at the tool list. Start by looking at route choice and project state.

The files that matter most

  1. AGENTS.md
  2. research-map.md
  3. findings-memory.md
  4. material-passport.yaml
  5. evidence-ledger.yaml
  6. logs/project-state/current.json
  7. logs/quality-gates/pipeline-status.json

For many non-technical users, the first four are the ones you will read most often.

Use cases

  • Literature review: review question, candidate literature, formal reference checks, synthesis
  • Social-platform or web case study: visible material, provenance, evidence capture, preparation for later coding
  • Computational social science project: literature, materials, analysis, writing, reproducibility
  • Writing and submission: drafting, citation checks, writing quality, revision, submission materials

Integrations

In simple terms:

  • Zotero holds formal references
  • Obsidian holds longer-lived notes
  • the project folder holds runtime state and traces

For platform and web cases, the repository stays conservative: prefer browser-visible material, keep provenance explicit, and keep capture separate from interpretation.

The product rule is simple: make the source visible enough that another person can check where the evidence came from.

Repository and future app

The simplest way to think about the relationship is:

  • this repository is the rules and documentation layer
  • the future app is the easier daily interface

The product idea is simple: the repository defines the workflow package; an app can make that workflow easier to start, inspect, and continue.

The repository is the source, the app is the interface.

If something goes wrong, where to look first

  1. Where is the project blocked? Look at logs/project-state/current.json.
  2. What is the project trying to do? Look at research-map.md and findings-memory.md.
  3. Did a checkpoint fail? Look at logs/quality-gates/pipeline-status.json.
  4. Only then suspect the repository itself: run python .\scripts\validate_research_stack.py.
