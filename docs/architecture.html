Architecture

How the system is organized underneath the surface.

If you already understand what the repo is trying to do, this page explains how it hangs together: how tasks are interpreted, how project work is organized, how checks block weak work, and how verified material moves into the right place.

Back to home Operator Guide

Think of it as four practical layers

  1. Task choice: `research-autopilot` explains what kind of work this is and how to begin.
  2. Project coordination: `research-team-orchestrator` turns larger work into roles, reviews, and handoffs.
  3. Checks: citations, writing quality, stage transitions, and reproducibility can block weak work.
  4. Knowledge handoff: verified material moves into Zotero, Obsidian, and project files.

Why the multi-agent layer matters

This repo does not count simple role-play as multi-agent work. It only counts as real multi-agent work when roles, outputs, reviews, and logs are actually separated. That is what makes a project readable later.

The files that matter most

  • `.codex/dispatch/`
  • `.codex/context-packets/`
  • `outputs/agent-runs/`
  • `logs/agent-handoffs/`
  • `logs/quality-gates/`
  • `logs/project-state/`

These files exist for a simple reason: when something goes wrong, a human should still be able to see where the project got stuck and what happened before that.
